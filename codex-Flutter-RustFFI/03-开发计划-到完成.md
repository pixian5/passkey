# 03 开发计划（直到完成）

## Phase 0：基线清理（1~2天）
- [ ] 拆分 `apps/app_flutter/lib/main.dart` 为模块化结构。
- [ ] 保留现有功能行为不变，补充最小回归测试。
- [ ] 统一日志与错误提示。

## Phase 1：FFI 骨架（2~4天）
- [ ] 在 `pass-core-ffi` 建立统一导出规范（init/shutdown/health）。
- [ ] 建立 Dart 侧动态库加载器与符号绑定。
- [ ] 完成最小闭环：`version`、`time_range_compare`、`ping`。

## Phase 2：账号核心下沉（4~7天）
- [ ] Rust 实现账号 CRUD API（含回收站语义）。
- [ ] Flutter 切换到 FFI API，不再直接操作 JSON 结构。
- [ ] 保留兼容导入器，迁移旧 `state.json`（若存在）。

## Phase 3：别名与导出下沉（3~5天）
- [ ] 将别名合并算法迁入 Rust。
- [ ] 将 CSV 导入导出迁入 Rust（复用 `pass-csvio`）。
- [ ] Flutter 仅负责文件选择与进度展示。

## Phase 4：同步协议联调（5~10天）
- [ ] 对接 `pass-transport` 协议对象（先本地 mock）。
- [ ] Pull/Push 冲突处理链路打通。
- [ ] 完成断网重试和错误恢复策略。

## Phase 5：稳定化与交付（3~5天）
- [ ] Windows/Ubuntu 打包构建脚本。
- [ ] 端到端验收清单全部通过。
- [ ] 完成 README/运维文档/已知问题。

## 关键里程碑
- M1：FFI 基线可用
- M2：账号完整能力从 Dart 迁移到 Rust
